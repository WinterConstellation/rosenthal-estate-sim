# 도입부 공포 UI 방향 의견 메모

목적: 다른 GPT/코덱스와 초반 UI, 선택지 색상, 낮/밤 연출 방향을 검토하기 위한 핑퐁용 의견 메모.

이 문서는 확정 규칙이 아니다.
코덱스가 바로 구현 지시로 받아들이기보다, 현재 구현과 비교하면서 의견을 주고받기 위한 초안이다.

## 현재 의견

도입부에서 빨간 선택지가 너무 많으면 유저가 위험 신호를 배우기 전에 혼란을 느낄 가능성이 있다.
개인 의견으로는 빨간색을 일반 선택지 색이 아니라, 금기/위험/돌이킬 수 없음/제물/피와 연결된 특별 신호로 아껴 쓰는 편이 좋아 보인다.

초반은 겉으로는 평화로운 영지 관리처럼 진행하고, 공포는 문장과 상황의 위화감으로 가볍게 암시하는 편이 효과적일 가능성이 높다.
유저가 먼저 정상적인 낮의 흐름을 익힌 뒤, 그 정상성이 조금씩 무너질 때 공포 효과가 커진다는 판단이다.

## 초반 낮 방향 제안

- 낮에는 자원, 사람, 영지 관리가 중심인 편이 좋다.
- 공포도, 이상현상, 침식 같은 수치는 초반에는 기본적으로 숨기는 쪽을 검토한다.
- 내부 계산은 유지하되, 유저에게는 숫자 대신 분위기 문장으로만 암시하는 편이 좋아 보인다.
- 예: "저택은 조용하다", "창고 쪽 공기가 눅눅하다", "하인들이 특정 방을 피한다".
- 선택지는 대부분 평범한 색을 사용하는 편이 안정적이다.
- 불길한 선택지도 초반에는 문구로만 암시하고, 색은 일반 선택지와 크게 다르게 만들지 않는 쪽을 우선 검토한다.

## 낮 글자/빛무리 연출에 대한 의견

현재 낮 화면에 떠다니는 글자 계열 연출은, 공포 텍스트라기보다 빛무리/먼지 입자/햇빛 반사에 가까운 의도로 보인다.
따라서 낮의 글자 연출을 곧바로 "공포 UI"로 분류해서 끄거나 약화할 필요는 없어 보인다.

다만 함수명이나 변수명이 `textRain`, `HorrorTextRain`, `rainGlyphs`처럼 되어 있으면, 다른 GPT/코덱스가 "낮에도 공포 글자비가 내린다"고 오독할 가능성이 있다.
의견으로는 이름을 조금 바꾸는 편이 안전하다.

후보:

- `HorrorTextRainCanvas` → `AmbientGlyphCanvas`
- `textRain` → `ambientGlyphs`
- `rainGlyphs` → `glyphParticles`
- `generateRain` → `generateGlyphField`
- `HORROR_TEXT_RAIN_GLYPHS` → `AMBIENT_GLYPHS`
- `HORROR_TEXT_RAIN_SEED` → `AMBIENT_GLYPH_SEED`
- `horror-text-rain-canvas` → `ambient-glyph-canvas`

톤 이름도 단순히 `day`/`night`보다 다음처럼 나누면 의도가 더 선명할 수 있다.

- 낮: `day-glow`
- 밤/지하: `night-rain`

주석 후보:

```js
// day-glow는 공포 UI가 아니라 낮 화면의 빛무리다.
// 유저에게 위험 수치나 경고로 읽히면 안 된다.
// 밤/지하에서는 같은 글리프 계열이 붉은 텍스트 비로 변질될 수 있다.
```

## 빨간 선택지 운용 의견

- 도입부와 초반 낮에는 빨간 선택지를 0~1개 수준으로 제한하는 편이 좋아 보인다.
- 빨간 선택지는 정말 특별한 선택에만 사용하는 쪽을 제안한다.
- 사용 후보: 피, 제물, 금기, 돌이킬 수 없는 선택, 명백한 위험, 밤/지하와 직접 연결된 선택.
- 단순히 수상하거나 분위기가 어두운 선택에는 빨간색을 쓰지 않는 편이 좋다.
- 빨간색이 많아지면 위험 신호가 아니라 기본 UI처럼 보일 수 있다.

## 공포 수치 표시 의견

- 낮 기본 상태: 공포/이상현상 수치 표시를 숨기는 쪽을 검토한다.
- 낮 이상 징후 낮음: 짧은 분위기 문장만 표시한다.
- 낮 이상 징후 중간: UI 일부가 흐리게 반응하거나 텍스트가 불안정해지는 정도를 검토한다.
- 낮 이상 징후 높음: 공포/이상현상 패널을 제한적으로 켜는 방향이 가능하다.
- 밤, 지하, 특정 이벤트, 임계치 초과, 진실 발견 이후에는 수치 또는 위험 표시를 활성화할 수 있다.

## 진행 리듬 의견

초반 낮: 평화로운 관리 + 약한 암시.
초반 밤: 처음으로 빨간 선택지와 공포 UI의 의미를 보여주는 후보 구간.
중반 이후: 공포/이상현상이 낮 화면에도 침투하기 시작.
후반: 낮/밤 경계가 약해지고, 숨겨진 수치가 UI에 직접 영향을 주기 시작.

## 구현 검토 시 주의 의견

- 공포 수치를 숨기는 것은 계산을 멈추는 뜻이 아니다. 내부 상태는 계속 누적된다.
- 유저가 확률이나 위험을 판단해야 하는 시점에는 충분한 신호를 줘야 한다.
- 다만 초반에는 숫자보다 문장, 색보다 분위기, 노골적 경고보다 위화감을 우선하는 편이 좋아 보인다.
- 빨간색은 시스템 언어로 보존하는 쪽이 효과적이다. 많이 쓰면 의미가 사라질 수 있다.
- 낮 빛무리는 유지해도 되지만, 이름이 공포 UI처럼 읽히면 후속 작업자가 잘못 고칠 위험이 있다.

## 결론 의견

도입부는 평화롭게 보이되 완전히 안전하게 보이면 안 된다.
초반의 목표는 공포를 설명하는 것이 아니라, 나중에 망가질 정상 상태를 유저에게 먼저 체감시키는 것에 가깝다.

따라서 현재 의견은 다음이다.

- 낮 빛무리는 유지 가능.
- 낮 빛무리를 공포 UI로 분류하지 않는 편이 좋음.
- 빨간 선택지는 초반에 아껴 쓰는 편이 좋음.
- 문서와 코드 명명은 확정 규칙처럼 보이지 않게 조정하는 편이 좋음.

코덱스는 위 내용을 확정 구현 지시로 처리하지 말고, 현재 코드와 비교한 뒤 반론 또는 대안 구현안을 제시할 것.
