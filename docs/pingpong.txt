# 핑퐁 파일 정리 및 실제 수정 이력

이 문서는 최신 작업 인계용이다.
짧게 압축하지 않는다.
작업자가 다음 흐름을 놓치지 않도록 현재 정본 상태, 완료된 구조, 남은 문제, 다음 작업 기준을 함께 남긴다.

## 최신 기준

- 기준 원격: `origin/main`
- 기준 시각: 2026-06-21 KST

## 작업 해석 기준

- `수정해`, `고쳐`, `반영해`, `넣어`, `정리해`, `진행해`는 실행 지시다.
- `검토만`, `수정하지 마`, `핑퐁 X`, `커밋하지 마`, `아직 하지 마`가 명시된 경우에만 쓰기 작업을 멈춘다.
- 사용자의 최신 입력을 임의로 보류문이나 반대 의미로 바꾸지 않는다.
- 애매하면 현재 파일과 이 문서를 읽고 작업 범위를 줄여 실행한다.
- 쓰기 전 최신 파일 SHA를 확인한다.
- 409가 나면 덮어쓰지 말고 최신 파일을 다시 읽는다.
- 작업 후 변경 파일, 검증, 빌드 여부를 보고한다.
- 핑퐁 외 새 인계 파일을 만들지 않는다.
- 현재 커밋 순서는 논의 순서와 완전히 일치하지 않을 수 있다. 커밋 순서만 보고 정본을 판단하지 말고, 최신 `main`의 파일 내용과 이 핑퐁의 정리 기준을 우선한다.

## 현재 용어 정리 기준

- `페이지`는 회차/반복 기록 번호를 뜻한다.
- `일차`는 날짜 진행을 뜻한다.
- `오전`/`오후`/`저녁`/`밤`은 시간대를 뜻한다.
- `페이즈`는 현재 화면/진행 상태를 뜻한다.
- `턴`은 내부 처리 단위다. 사용자 UI 노출은 최소화한다.
- `세션`은 개발/저장/검증 용어다. 본편 UI에는 노출하지 않는다.

최종 UI 계층:

```txt
상단 좌표: n페이지 · n일차 · 시간
화면 성격: 영지 업무 / 지하 탐사 / 선택의 결과 / 기록 확인
```

정리 기준:
- `페이지`는 상단 좌표에서만 회차 단위로 사용한다.
- `headerTitle`이나 화면 제목에서 `지하 페이지`, `영지 페이지`처럼 다시 쓰지 않는다.
- `페이즈`라는 단어 자체는 일반 플레이 화면에 직접 노출하지 않는다.
- 화면에는 `영지 업무`, `지하 탐사`, `선택의 결과`, `기록 확인`처럼 자연어 페이즈명만 보인다.

현재 코드 상태와 목표 정리:
- 현재 `main`에는 기존 UI 보정 작업과 page marker helper가 들어가 있다.
- `headerTitle`은 `영지 업무`/`지하 탐사`/`선택의 결과`/`기록 확인` 계열로 정리했다.
- verify와 작업 로그도 같은 기준으로 맞췄다.

## 현재 확정 상태

- 성흔/낙인 표식은 총 `100개`가 정본이다.
  - 성흔 50개
  - 낙인 50개
  - 성향형 94개
  - 단독 6개
- `MARK_BRANCH_UNLOCKS`는 현재 총량 안에서 모두 도달 가능하다.
- 기존 저장 마이그레이션(`LEGACY_STIGMA_MARK_MAP`)은 유지한다.
- UI 단계는 `day-calm`, `day-anomaly`, `day-critical`, `night-calm`, `night-anomaly`, `night-critical` 6단계다.
- 기본 밤은 푸른 `night-calm`이며 붉은 밤은 `night-anomaly` 이후에 사용한다.
- 공포 UI는 장식이 아니라 실제 낮/밤, 이상 단계, 공포 상태와 연결되는 표시 체계다.

## 완료된 구조

### 콘텐츠 레지스트리

- `src/engine/storyletRegistry.js`를 추가했다.
- `triggerKey`, 조건, 우선순위, 반복 여부 기반 후보 색인을 만든다.
- `src/engine/dayActionStorylets.js`를 추가했다.
- `DAY_ACTIONS`를 `day-category:*` storylet으로 등록했다.
- `getDayOffers`는 낮 행동 후보를 직접 전체 배열에서 필터링하지 않고 category triggerKey로 좁힌 후보를 사용한다.
- `src/engine/explorationStorylets.js`를 추가했다.
- `EXPLORATION_EVENTS`를 `exploration-direction:*` storylet으로 등록했다.
- `startExpedition`은 direction별 직접 배열 필터링 대신 direction triggerKey 후보를 사용한다.
- `SPECIAL_EVENT_GROUPS`는 `src/data/scriptPacks/specialEventGroups.js`로 분리했다.
- `src/data/scriptManifest.js`는 `special-event-groups` pack의 id, triggerKey, exportName, 수량 메타를 가진다.
- `src/engine/scriptLoader.js`는 manifest 기준으로 특수 사건 본문 pack을 dynamic import한다.
- `App.jsx`는 `special-event` phase에 진입했을 때 특수 사건 본문 pack을 필요 시점에 불러온다.

### 엔진 책임선 1차

- `src/engine/rosenthalEngine.js`는 본편 상태 생성, phase 전환, 낮/밤 진행, 저장용 run/meta 동기화의 정본 엔진이다.
- `src/engine/rulesEngine.js`는 확률, 수치 보정, 선택 결과 계산, 직업/표식/패시브 조회 중심으로 줄였다.
- 본편 미사용 구형 진행 API는 `src/engine/legacyProgressionEngine.js`로 이동해 보존했다.
- `rulesEngine.js`는 더 이상 `tutorialRules.js`의 구형 낮/밤 콘텐츠를 직접 import하지 않는다.

### 화면 분리 1차

- `src/components/DialogueCard.jsx`를 추가했다.
- 대화 카드와 `normalizeDialogue`를 공유 컴포넌트/표시 유틸로 분리했다.
- `src/screens/ResultOverlay.jsx`를 추가했다.
- 결과 대화, 결과 카드, 변경 목록, 알림, 확인 버튼 표시를 App 밖으로 옮겼다.
- `App.jsx`는 기존처럼 하단에서 `<ResultOverlay ... />`를 호출한다.
- `mainContent`에 result phase branch를 새로 만들지 않았다.
- `continueAfterResult`, `completeTransition`, 엔진 파일, 결과 문구, 판정, 저장 구조는 변경하지 않았다.

### 프롤로그/1일차 톤

- 프롤로그는 따뜻한 환대와 낯선 준비 상태를 중심으로 낮췄다.
- 불길함은 서류, 빈 의자, 너무 정확한 준비 같은 생활 불일치로만 남긴다.
- 1일차 낮은 카테고리별 `firstDaySafe` 후보를 우선 사용한다.
- 단순 비용, 피로, 약한 불길함은 `warning`/`uneasy` 계열로 보인다.
- 실제 강한 선택지만 `danger` 이상으로 둔다.
- 엔진의 선택 결과 계산과 자동 플레이 smoke 검증은 약화하지 않았다.

### 첫날 힌트 팝업

- `src/components/FirstDayHintModal.jsx`를 추가했다.
- 첫날 낮 최초 진입 시 `firstDayHintOpen` 화면 상태로 표시한다.
- 닫으면 `firstDayHintSeen: true`, `firstDayHintVariant: "ledger-v2"`를 기록한다.
- 이전 저장/테스트에서 `firstDayHintSeen`이 true였더라도 새 `ledger-v2` 인계서 팝업은 한 번 다시 뜬다.
- 힌트는 낮 업무, 선택에 따른 수치 변화, 해가 진 뒤의 진행만 안내한다.
- 문구는 로젠탈 저택이 새 영주에게 건네는 3단계 인계서처럼 처리한다.
- 세계 규칙의 정답은 설명하지 않는다.
- 자동 플레이 smoke는 UI를 렌더링하지 않으므로 기존처럼 통과한다.

### 자동 플레이 smoke 검증

- `scripts/verify-game.mjs`에 `KNOWN_AUTOPLAY_PHASES`와 `autoplayHandlers`를 추가했다.
- 실제 App 흐름과 같은 `seed-reveal -> prologue -> day` 진입을 검증한다.
- 자동 플레이는 첫 번째 유효 선택지만 고르는 단순 순찰 검증이다.
- `result`의 `pendingResult`, 알 수 없는 phase, NaN/Infinity, 반복 key 정체를 실패로 처리한다.
- 최소 5개 seed를 두 번씩 실행해 `record-stop` 또는 `ending` 도달과 projection 결정성을 확인한다.
- `App.jsx`, `rosenthalEngine.js`, `rulesEngine.js`는 이 작업에서 수정하지 않았다.

### 글리프 분위기 효과

- `src/effects/GlyphAtmosphereCanvas.jsx`와 `src/effects/glyphAtmosphere.css`를 삭제했다.
- `src/App.jsx`의 import, director 상태, `glyphFormat`, Canvas 렌더링 경로를 제거했다.
- 공포 오버레이는 글자 안개, 정적 문장, 텍스트 눈동자만 사용한다.

### 표식 도감과 분기 진행도

- 모든 표식은 정적 데이터인 `codexText`, `sourceHint`를 가진다.
- 두 필드는 효과 계산과 저장 데이터에 관여하지 않는다.
- 보유 표식은 도감 문장, 미획득 표식은 획득 힌트, 미해금 표식은 수집 조건을 표시한다.
- `getMarkBranchProgress(markIds)`는 요구 조건별 현재값, 요구값, 남은 수량, 해방 여부를 반환한다.
- `getMarkBranchLabel(branchKey)`는 수량 분기와 성향별 capstone 분기에 사용자용 이름을 제공한다.
- 장비/도감 화면은 해방된 분기와 성흔·낙인·혼합 축의 다음 목표를 최대 1개씩 표시한다.

## 완료: UI/표기/수치/카운터 보정

2026-06-21 기준 완료했다.
새 콘텐츠 확장이 아니라 현재 플레이 화면의 의미 오류를 바로잡는 작업이다.

### 1. DialogueCard 텍스트 계층 분리

현재 문제:
- 턴 정보는 상단에 이미 있다.
- 카드 안에서 현재 턴 제목, 화자, 본문이 한 덩어리처럼 붙어 보인다.
- 읽는 순서가 흐려진다.

수정 방향:
- 상단 턴 정보는 유지한다.
- 카드 내부는 아래 계층으로 분리한다.

```txt
현재 턴의 제목
────────────────
화자
본문
```

구현 기준:
- `DialogueCard.jsx`와 CSS 중심으로 수정한다.
- `ChoicePanel` 전체를 뜯지 않는다.
- `eyebrow`는 상단 보조 정보로 유지한다.
- `title`은 본문 제목으로 분리한다.
- speaker label은 화자로 분리한다.
- 선택지 목록 UI와 첫날 힌트 팝업은 이번 수정에 끌어들이지 않는다.

Acceptance:
- 제목, 화자, 본문이 시각적으로 분리된다.
- 기존 대화 넘김과 continue 동작은 유지된다.
- `npm.cmd run verify`, `npm.cmd run build` 통과.

### 2. 페이지/일차/시간 표기 보정

현재 문제:
- 상단 `dream-mark`가 `{game.day}번째 꿈 - {game.day}번째 밤`처럼 날짜와 묶여 있다.
- 꿈 번호와 밤 번호가 둘 다 `day` 기준이라 회차/기록 축을 표현하지 못한다.
- `n번째 꿈`, 한글 서수 꿈 표기, 장문 서수 꿈 표기는 UI 좌표로 부적합하다.

수정 방향:
- 최종 UI 표기는 `1페이지 · 1일차 · 오전`, `1페이지 · 1일차 · 밤`, `129페이지 · 3일차 · 밤` 형식을 사용한다.
- `페이지`는 초반 꿈처럼 보이는 반복 경험과 후반 실존 경험 전환을 모두 담을 수 있는 기록 단위로 사용한다.
- page 번호는 `day`가 아니라 실제 회차/기록 축에서 가져온다.
- 실제 증가하는 값이 `game.cycle`인지 `game.meta.cycle`인지 확인한다.
- 낮/밤/오전/오후/저녁 표기는 현재 phase와 `dayTurn` 기준으로 유지한다.
- `꿈`이라는 단어를 직접 번호로 세지 않는다. 꿈의 의미는 페이지/기록 구조에 묻히게 둔다.

Acceptance:
- 상단 표기가 `n페이지 · n일차 · 시간` 형식으로 바뀐다.
- 다음 회차에서 page 번호가 바뀐다.
- `n번째 꿈`/`n번째 밤` 계열 문구가 App 표시 코드에서 사라진다.
- 낮/밤 날짜 표기는 깨지지 않는다.
- 가능하면 표시 helper 테스트를 verify에 추가한다.

### 3. 성향 레벨의 음수 변화 보정

현재 문제:
- 성향 레벨이 높을수록 보너스뿐 아니라 음수 변화도 커지는 것으로 보인다.
- 성장 보상이 손해를 키우는 느낌을 줄 수 있다.

수정 방향:
- 성향 레벨 보정은 긍정 효과에 우선 적용한다.
- 음수 delta에 같은 배율을 적용하는 구조는 재검토한다.
- 특정 빌드가 특정 감소값을 더 크게 받는 의도라면 별도 규칙으로 명시한다.

Acceptance:
- 성향 레벨 상승이 단순 음수 변화 증폭으로 작동하지 않는다.
- verify에 양수/음수 delta 배율 차이를 검사한다.
- 기존 직업/표식/패시브 계산과 충돌하지 않는다.

### 4. 역방향 수치 색상 보정

현재 문제:
- 일부 수치는 증가할수록 좋지 않은 변화다.
- 이런 값의 `+`가 일반 긍정 증가처럼 보이면 안 된다.

수정 방향:
- change display에서 key 또는 group 기준으로 polarity를 구분한다.
- `fear`, `corruption`, `missing`, `horrorTraits`, `derivedHorror` 계열은 증가가 나쁜 변화다.
- 나쁜 증가는 danger 계열 색으로 표시한다.
- 좋은 감소는 완화색으로 표시한다.
- 단순히 delta 부호만으로 positive/negative class를 정하지 않는다.

Acceptance:
- 공포 +, 이상징후 +, 공포 특성 +가 위험색으로 보인다.
- 체력/식량/신뢰 같은 일반 좋은 값 증가는 기존 긍정색을 유지한다.
- `getChangeDetail` 또는 result change 표시가 polarity를 반영한다.

### 5. 확정 선택 표시 보정

현재 문제:
- 일부 선택은 확률 판정이 아니라 확정 선택이어야 한다.
- 일반 확률 보정에 휘둘리면 의미가 흐려진다.

수정 방향:
- `leave-companion` 계열 선택은 성공률 100%로 고정한다.
- preview/detail도 성공률 100%로 보여야 한다.
- 성향, 직업, 표식, 패시브가 이 성공률을 바꾸지 않는다.
- 이 선택의 무게는 실패 확률이 아니라 영구 결과와 비용이다.

Acceptance:
- 해당 선택은 항상 성공률 100%로 표시된다.
- 관련 보정값이 이 확률을 바꾸지 않는다.
- verify에 성공률 고정 검사를 추가한다.

### 6. 진행 카운터 표시 보정

현재 문제:
- 카운터가 `■■ 0 / 3`처럼 가려진다.
- 첫 누적 이후에도 완전히 가려져 있으면 피드백이 부족하다.

수정 방향:
- 0일 때는 은폐 표시를 유지할 수 있다.
- 1 이상부터는 `제물 1 / 3` 계열로 의미가 드러나야 한다.
- 저장/로드 후에도 표시가 유지되어야 한다.
- 관련 선택 후 카운터가 즉시 바뀐다.

Acceptance:
- count 0일 때는 은폐 표시 가능.
- count 1 이상이면 의미가 드러난다.
- 관련 선택 후 카운터가 즉시 바뀐다.

완료 확인:
- `src/components/runMarkers.js`에 page marker, 제물 카운터, 변화량 polarity helper를 분리했다.
- App 상단 `dream-mark`와 밤 진입 eyebrow는 page marker helper를 사용하며 `App.jsx` 표시 코드에서 `n번째 꿈`/`n번째 밤` 계열 문구를 제거했다.
- `headerTitle`은 페이지 번호를 반복하지 않는 페이즈 표시로 정리했다. 최종 기준은 낮 `영지 업무`, 밤 `지하 탐사`, 결과 `선택의 결과`, 기타 `기록 확인`이다.
- `phaseLabel`에서도 단독 `페이지` fallback을 제거했고, `페이지`는 상단 좌표 표기에만 남긴다.
- `src/components/runMarkers.js`의 phase title/label helper와 verify 검증을 같은 기준으로 맞췄다.
- `DialogueCard`는 제목/화자/본문 계층을 CSS와 함께 분리했다.
- 성향 레벨 보정은 양수 능력치 변화에만 적용하고, 음수 변화는 배율을 타지 않는다.
- 결과 변화 색상은 단순 부호가 아니라 `fear`, `corruption`, `missing`, 공포 특성 계열의 역방향 의미를 반영한다.
- `leave-companion` 계열 선택은 실제 판정 경로까지 `fixedChance`를 통과시켜 성공률 100% 고정으로 처리한다.
- 제물 카운터는 0일 때 은폐 표시, 1 이상부터 `제물 n / 3`로 공개한다.
- `npm.cmd run verify`, `npm.cmd run build` 통과.

## 완료: 본문 지연 로딩 실험

원격 핑퐁의 UI/표기/수치/카운터 보정 지시는 유지한다.
다만 로컬 작업트리에서 특수 사건 본문 1개 pack 기준 loader 실험은 먼저 완료했다.

- `SPECIAL_EVENT_GROUPS`를 `src/data/scriptPacks/specialEventGroups.js`로 옮겼다.
- `src/data/scriptManifest.js`에 `special-event-groups` manifest를 추가했다.
- `src/engine/scriptLoader.js`가 manifest 기준으로 특수 사건 본문 pack을 dynamic import한다.
- `App.jsx`는 `special-event` phase에서 본문 pack을 필요할 때 로드하고, 로드 전에는 대기 UI를 표시한다.
- 자동 플레이 smoke 검증은 loader 경로로 특수 사건 본문을 불러와 기존 phase 진행을 확인한다.
- `npm.cmd run verify`, `npm.cmd run build` 통과를 확인했다.

## 완료: 로컬 스크립트 편집 인덱스 1차 구현

- Understand Anything은 코드베이스 시각화 전용으로 유지한다.

- 실행 명령은 `npm.cmd run script-edit:index`, `npm.cmd run script-edit`이다.
- 첫 버전은 특수 사건 본문과 manifest 필드 편집을 대상으로 한다.
- 저장은 allowlist, source hash, source-span locator 검사를 통과해야 한다.
- `.script-edit/index.json`과 backups는 생성물로 Git에 올리지 않는다.
- 확인: `npm.cmd run verify`, `npm.cmd run build`

## 파일 작성 규칙

- 텍스트 파일은 UTF-8(BOM 없음)·LF를 유지한다.
- Windows PowerShell의 `>`, `>>`, `Out-File`로 소스/문서를 작성하지 않는다.
- 수정은 패치 또는 UTF-8을 명시하는 도구를 사용하며, 전수 인코딩 검사는 이상 징후가 있을 때만 수행한다.
