# 핑퐁 파일 정리 및 실제 수정 이력

이 문서는 최신 작업 인계용이다.
길어지면 이전 논의와 폐기안을 지우고, 다음 작업자가 바로 이어받을 현재 판단과 확인 항목만 남긴다.

## 최신 기준

기준 원격: `origin/main`
기준 시각: 2026-06-20 KST

## 현재 확정 상태

- 성흔/낙인 표식은 총 `100개`가 정본이다.
  - 성흔 50개
  - 낙인 50개
  - 성향형 94개
  - 단독 6개
- `MARK_BRANCH_UNLOCKS`는 현재 총량 안에서 모두 도달 가능하다.
- 기존 저장 마이그레이션(`LEGACY_STIGMA_MARK_MAP`)은 유지한다.
- UI 단계는 `day-calm`, `day-anomaly`, `day-critical`, `night-calm`, `night-anomaly`, `night-critical` 6단계다.
- 기본 밤은 푸른 `night-calm`이며 붉은 밤은 `night-anomaly` 이후에 사용한다.

## 이번 작업 의도

- 방어적인 직접 배열 필터링에서 공격적인 storylet 레지스트리 흐름으로 1차 전환한다.
- 낮 행동과 밤 탐사 이벤트를 triggerKey 색인 기반 실제 진행 경로에 연결한다.
- `rosenthalEngine.js`를 본편 진행 엔진으로 두고, `rulesEngine.js`는 계산 규칙 중심으로 줄인다.
- `App.jsx`의 결과 오버레이 표시를 독립 컴포넌트로 분리한다.
- 다음은 자동 플레이 smoke 검증으로 phase 진행성을 지킨다.
- 글리프 제거, 표식 도감 문장, 분기 진행도 구조는 현재 정본으로 유지한다.

## 완성된 구조

- 콘텐츠 레지스트리:
  - `src/engine/storyletRegistry.js`를 추가해 `triggerKey`, 조건, 우선순위, 반복 여부 기반 후보 색인을 만들었다.
  - `src/engine/dayActionStorylets.js`를 추가해 `DAY_ACTIONS`를 `day-category:*` storylet으로 등록했다.
  - `getDayOffers`는 이제 낮 행동 후보를 직접 전체 배열에서 필터링하지 않고, category triggerKey로 좁힌 후보만 사용한다.
  - `src/engine/explorationStorylets.js`를 추가해 `EXPLORATION_EVENTS`를 `exploration-direction:*` storylet으로 등록했다.
  - `startExpedition`은 이제 `EXPLORATION_EVENTS` 직접 direction 필터링 대신 direction triggerKey 후보를 사용한다.
  - 아직 본문 지연 로딩은 적용하지 않았다. 이번 단계는 실제 진행 경로에 레지스트리 골격을 먼저 꽂는 것이다.
- 엔진 책임선 1차:
  - `src/engine/rosenthalEngine.js`는 본편 상태 생성, phase 전환, 낮/밤 진행, 저장용 run/meta 동기화의 정본 엔진이다.
  - `src/engine/rulesEngine.js`는 확률, 수치 보정, 선택 결과 계산, 직업/표식/패시브 조회 중심으로 줄였다.
  - 본편 미사용 구형 진행 API(`createInitialGame`, `getNightOffers`, `getTitles`, `hasPeacefulLordEnding` 등)는 `src/engine/legacyProgressionEngine.js`로 이동해 보존했다.
  - `rulesEngine.js`는 더 이상 `tutorialRules.js`의 구형 낮/밤 콘텐츠를 직접 import하지 않는다.
- 화면 분리 1차:
  - `src/components/DialogueCard.jsx`를 추가해 대화 카드와 `normalizeDialogue`를 공유 컴포넌트/표시 유틸로 분리했다.
  - `src/screens/ResultOverlay.jsx`를 추가해 결과 대화, 결과 카드, 변경 목록, 알림, 확인 버튼 표시를 App 밖으로 옮겼다.
  - `App.jsx`는 기존처럼 하단에서 `<ResultOverlay ... />`를 호출하며, `mainContent`에 result phase branch를 새로 만들지 않았다.
  - `continueAfterResult`, `completeTransition`, 엔진 파일, 결과 문구, 판정, 저장 구조는 변경하지 않았다.
- 글리프 분위기 효과:
  - `src/effects/GlyphAtmosphereCanvas.jsx`와 `src/effects/glyphAtmosphere.css`를 삭제했다.
  - `src/App.jsx`의 import, director 상태, `glyphFormat`, Canvas 렌더링 경로를 제거했다.
  - 공포 오버레이는 글자 안개, 정적 문장, 텍스트 눈동자만 사용한다.
- 표식 도감:
  - 모든 표식은 정적 데이터인 `codexText`, `sourceHint`를 가진다.
  - 두 필드는 효과 계산과 저장 데이터에 관여하지 않는다.
  - 보유 표식은 도감 문장, 미획득 표식은 획득 힌트, 미해금 표식은 수집 조건을 표시한다.
- 분기 진행도:
  - `getMarkBranchProgress(markIds)`는 요구 조건별 현재값, 요구값, 남은 수량, 해방 여부를 반환한다.
  - `getMarkBranchLabel(branchKey)`는 수량 분기와 성향별 capstone 분기에 사용자용 이름을 제공한다.
  - 장비/도감 화면은 해방된 분기와 성흔·낙인·혼합 축의 다음 목표를 최대 1개씩 표시한다.

  - `MARKS=100`, 성흔 50, 낙인 50, 분기 조건 도달 가능성
  - 모든 표식의 `codexText`, `sourceHint`

  - 삭제된 글리프 파일과 `App.jsx` 글리프 참조 부재
  - 낮 행동 storylet 수량, triggerKey, category별 후보 순서, `requiresFlag` 조건
  - 밤 탐사 storylet 수량, triggerKey, direction별 후보 순서, 고정 seed `startExpedition` eventIds 유지
  - `rulesEngine.js` 구형 진행 API export 부재와 `legacyProgressionEngine.js` 보존 API 존재
  - `App.jsx` 내부 `ResultOverlay`/`DialogueCard` 함수 부재와 독립 컴포넌트 export 존재

  - 장비/도감에서 보유·미획득·미해금 문구가 구분되는지

  - 글리프 없이 글자 안개·정적 문장·눈동자가 기존 조건에서 동작하는지

## 구조 판단 보고서 (남은 의도 확인)

아래 내용은 현재 코드와 일반적인 선택형 로그라이크 구조를 비교한 정적 분석이다.
제작자의 의도가 확인되기 전에는 확정 작업 방향으로 취급하거나 관련 코드를 제거하지 않는다.

### 1. 당장 추가를 검토할 것

- 대규모 콘텐츠용 최소 레지스트리: 낮 행동에 1차 적용했다. 다음은 적용 범위 확대와 본문 로더 실험이 필요하다.
- 정본 엔진 단일화: `rosenthalEngine.js`를 본편 엔진으로 확정하고, `rulesEngine.js`의 구형 진행 API는 `legacyProgressionEngine.js`로 분리했다.
- 저장 마이그레이션: `GAME_VERSION`별 변환 함수와 저장 손상 전 백업 1개를 둘지 확인해야 한다.
- 자동 플레이 검증: 여러 시드와 단순 행동 정책으로 phase 중단, 도달 불가능한 분기, NaN을 검사하는 시뮬레이터가 필요해 보인다.
- CI 검증 게이트: Pages와 Windows 빌드 전에 `npm run verify`를 실행하도록 할지 확인해야 한다.

### 2. 당장 무너질 수 있는 위험

- `App.jsx`가 phase 렌더링, 개발자 도구, 모달, 공포 연출을 함께 소유해 새 phase가 늘수록 결합도가 빠르게 커질 수 있다.
- `rosenthalEngine.js`와 `rulesEngine.js`의 큰 책임선은 1차로 분리했지만, 일부 계산 helper의 위치는 화면 분리와 자동 플레이 검증 뒤 다시 정리할 수 있다.
- `rosenthalContent.js`와 `tutorialRules.js`에 동명 콘텐츠가 병존해 어느 쪽이 정본인지 혼동될 수 있다.
- 현재 콘텐츠가 정적 import이므로 스크립트가 수십 MB가 되면 초기 로드와 파싱 비용이 커질 수 있다.
- 수동 저장 10칸이 전체 상태를 `localStorage`에 복제하므로 history와 상태가 커질 때 용량 초과나 저장 지연 위험이 있다.
- 표식 100개에 비해 효과 프로필 반복이 많아 수집 규모 대비 체감 차이가 작아질 수 있다.

### 3. 일반적인 게임보다 훌륭한 부분

- 난수가 run seed, 날짜, history, choice id 조합으로 재현 가능해 디버깅과 회차 재현에 유리하다.
- 주요 선택 계산과 상태 전이가 기존 상태를 직접 훼손하기보다 새 상태를 반환하는 방식으로 구성돼 있다.
- NPC의 사망, 실종, 변질, 처치, 유품이 제물과 분기에 연결되어 선택이 영구 서사 결과를 만든다.
- 현재 회차와 영구 `meta`가 분리되어 있고 성향 성장, 표식, 엔딩 기록이 다음 회차로 이어진다.
- 공포 UI가 단순 장식이 아니라 낮/밤, 이상, 임계 단계와 실제 공포 상태에 연결된다.
- 단일 검증 스크립트가 확률, 표식, 분기, 저장, phase 전환을 비교적 넓게 다룬다.

### 4. 제작 의도를 확인해야 하는 부분

- `legacyProgressionEngine.js`에 보존한 구형 진행 API를 장기 보존할지, `reference/`로 이동할지는 추후 결정이 필요하다.
- 새 회차 시드가 일반 난수가 아니라 버튼을 누른 시각의 초 `00~59`로 결정되는 것이 의도된 연출인지 확인이 필요하다.
- 로그라이크 구조에서 수동 저장 10칸이 기록 보존용인지, 세이브 스컴 허용용인지 확인이 필요하다.
- 성향, 직업, 칭호, 패시브, 휴대표식, 장착표식, 성인 시드가 모두 선택 계산에 겹치는데 어느 층이 빌드의 중심인지 확인이 필요하다.
- `docs/large_script_architecture_theory_for_codex.txt`의 레지스트리/지연 로딩 방향을 실제 구현 기준으로 채택할지 확인이 필요하다.

### 제작자 확인 질문

1. `legacyProgressionEngine.js`의 구형 진행 API는 장기 보존, `reference/` 이동, 삭제 중 어느 쪽인가?
2. 초 단위 시드 선택은 작품 설정과 연결된 의도된 규칙인가?
3. 수동 저장 10칸은 결과 되돌리기를 허용하는 설계인가?
4. 여러 성장축 중 플레이어가 빌드의 중심으로 인식해야 하는 것은 무엇인가?
5. 레지스트리 다음 적용 대상은 특수 사건, 밤 탐사 이벤트, 본문 loader 중 무엇인가?

### 1순위: 자동 플레이 smoke 검증 추가

이 작업은 자동 플레이 AI를 만드는 작업이 아니다.
목표는 `scripts/verify-game.mjs` 안에 엔진 smoke test를 추가해 새 구조가 들어와도 phase가 막히지 않고 최소 경로가 7일 근처까지 굴러가는지 확인하는 것이다.

핵심 원칙:
- UI를 렌더링하지 않는다.
- React, `App.jsx`, 화면 컴포넌트를 건드리지 않는다.
- 엔진 로직을 고쳐서 테스트를 통과시키지 않는다.
- 자동 플레이는 재미있게 플레이하지 않는다.
- 자동 플레이는 첫 번째 유효 선택지만 고르는 단순 순찰 검증이다.
- 새 phase가 추가될 때 전체 자동 플레이를 뜯어고치지 않도록 phase handler map 구조로 만든다.

작업 대상:
- `scripts/verify-game.mjs`
- 필요하면 `docs/work_log.txt`
- 필요하면 `docs/pingpong.txt`

금지:
- `src/App.jsx` 수정 금지
- `src/engine/rosenthalEngine.js` 수정 금지
- `src/engine/rulesEngine.js` 수정 금지
- 밸런스, 문구, 선택 결과, 저장 구조 변경 금지
- 테스트 통과를 위해 게임 규칙을 약하게 만들지 말 것

구현 방향:
- `verify-game.mjs`에 `KNOWN_AUTOPLAY_PHASES`와 `autoplayHandlers`를 둔다.
- 자동 플레이는 `createNewRun({ second, runRngSeed })`에서 시작한다.
- `seed-reveal`은 `openFirstDay(state)`로 넘긴다.
- `day`는 `getDayOffers(state)`의 첫 번째 유효 선택지를 `chooseDayAction(state, choice)`로 처리한다.
- `special-event`는 `getSpecialGroup(state)`의 현재 stage 첫 option을 `chooseSpecialEvent(state, option)`으로 처리한다.
- `result`는 `pendingResult` 존재를 확인한 뒤 `continueAfterResult(state)`를 호출한다.
- `nightfall-transition`은 `completeTransition(state)`를 호출한다.
- `night-companion`은 `getCompanionOffers(state)`에서 `alone`을 우선 선택하고 `selectCompanion(state, companionId)`를 호출한다.
- `night-direction`은 `getDirectionOffers()`의 첫 direction을 골라 `startExpedition(state, direction.id)`를 호출한다.
- `expedition`은 `getCurrentExplorationEvent(state)`와 `getExplorationOptions(event)`를 사용하고, `isExplorationOptionAvailable(state, option)`인 첫 option을 `chooseExplorationOption(state, event, option)`으로 처리한다.
- `finale`는 `getCurrentFinale(state)`와 `getFinaleOptions(state, finale)`의 첫 option을 `chooseFinaleOption(state, finale, option)`으로 처리한다.
- `keepsake-delivery`는 `deliverKeepsake(state, "family")`를 사용한다.
- `transformed-choice`는 `chooseTransformedFate(state, "spare")`를 사용한다.
- `escape-transformed-choice`는 `chooseEscapeTransformedFate(state, "spare")`를 사용한다.
- `day-eight`은 `finishVerticalSlice(state)`를 호출한다.
- `record-stop`, `ending`은 성공 종료 phase로 취급한다.

필수 invariant:
- phase는 문자열이어야 한다.
- 알 수 없는 phase는 실패한다.
- `day`, `dayTurn`, `history.length`는 유한한 값이어야 한다.
- `resources`, `estate`, `stats`, `horrorTraits`, `derivedHorror` 안에 `NaN` 또는 `Infinity`가 없어야 한다.
- `history`는 배열이어야 한다.
- `result` phase에는 `pendingResult`가 있어야 한다.
- 동일한 `phase/day/dayTurn/expedition.stepIndex/history.length` 조합이 지나치게 반복되면 무한 루프로 실패한다.
- 실패 메시지에는 seed, step, phase, day, dayTurn을 포함한다.

결정성 검사:
- 같은 `{ second, runRngSeed }`로 자동 플레이를 두 번 실행한다.
- `runStartedAt` 같은 시간값은 비교하지 않는다.
- projection에는 `phase`, `day`, `dayTurn`, `route`, `endingId`, `sacrificeCount`, `history` 핵심값, `resources`, `estate`, `stats`, `horrorTraits`, `derivedHorror`를 포함한다.
- 같은 seed의 projection은 완전히 같아야 한다.

최소 seed:
- `{ second: 0, runRngSeed: "autoplay-smoke-0" }`
- `{ second: 1, runRngSeed: "autoplay-smoke-1" }`
- `{ second: 7, runRngSeed: "autoplay-smoke-7" }`
- `{ second: 13, runRngSeed: "autoplay-smoke-13" }`
- `{ second: 59, runRngSeed: "autoplay-smoke-59" }`

- 기존 assert를 지우거나 약화하지 않는다.
- 기존 검증 의미를 바꾸지 않는다.

중요:
이 작업은 자동 플레이를 똑똑하게 만드는 일이 아니다.
이 작업은 다음 구조 변경 때 이렇게 말하기 위한 것이다.

> 새 phase를 추가했으면 handler를 추가하라.
> handler가 없으면 verify가 실패한다.

그게 목적이다.

### 2순위: 본문 지연 로딩 실험

새 대형 스크립트 파일 1개를 manifest + loader로 분리하는 실험은 자동 플레이 smoke 검증 뒤에 진행한다.
자동 플레이가 먼저 있어야 loader 작업 뒤 phase 진행이 깨졌는지 바로 잡을 수 있다.

### 3순위: 성흔/낙인 효과 밸런스

구조가 고정되기 전에 표식 효과를 많이 바꾸지 않는다.
효과 밸런스는 자동 플레이 검증 뒤에 진행한다.

## 파일 작성 규칙

- 텍스트 파일은 UTF-8(BOM 없음)·LF를 유지한다.
- Windows PowerShell의 `>`, `>>`, `Out-File`로 소스/문서를 작성하지 않는다.
- 수정은 패치 또는 UTF-8을 명시하는 도구를 사용하며, 전수 인코딩 검사는 이상 징후가 있을 때만 수행한다.
