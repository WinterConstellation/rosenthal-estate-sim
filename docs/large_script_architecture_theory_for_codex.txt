대용량 스크립트 기반 게임/비주얼노벨 코드 구조 이론 지시서
작성 목적: Codex가 프로젝트 구조를 이해하고, 대용량 스크립트/선택지 데이터를 코드와 분리하여 안정적으로 다루기 위한 설계 원칙만 정리한다.
주의: 이 문서는 구현 코드가 아니라 구조 이론과 작업 지침이다. Codex는 이 문서를 읽고 기존 프로젝트에 맞춰 별도 구현 계획을 세워야 한다.

============================================================
1. 핵심 목표
============================================================

이 프로젝트는 스크립트, 선택지, 이벤트 텍스트, 연출 데이터가 매우 커질 수 있다.
예상 규모는 코드와 콘텐츠 데이터를 합쳐 수십 MB 이상이다.

따라서 구조의 핵심 목표는 다음과 같다.

1. 엔진 코드는 작게 유지한다.
2. 대사/선택지/이벤트 본문은 코드가 아니라 데이터로 분리한다.
3. 스크립트 본문은 기본적으로 OFF 상태로 둔다.
4. 조건이 만족될 때만 필요한 스크립트 데이터를 ON 상태로 호출한다.
5. 모든 스크립트를 매번 검사하지 않는다.
6. 항상 가벼운 색인(manifest/index)만 먼저 검사한다.
7. 조건 통과 후에만 무거운 본문 데이터를 로드한다.
8. 출력 엔진은 로드된 데이터에 따라 텍스트, 선택지, 움직임, 깜빡임, 기타 효과를 적용한다.
9. 화면에 보이지 않는 스크립트는 실행되지 않아야 한다.
10. 프로젝트가 커져도 로딩, 조건 검사, 렌더링이 선형으로 무거워지지 않게 한다.

요약:
코드는 뼈대, 스크립트는 장서, manifest는 카드목록, 조건 게이트는 문지기, 렌더러는 무대 장치다.

============================================================
2. 기본 철학: 코드와 콘텐츠의 분리
============================================================

대용량 스크립트를 JavaScript/TypeScript 코드 안에 직접 넣으면 안 된다.
대사와 선택지는 코드가 아니라 외부 데이터로 둔다.

나쁜 방향:
- 모든 대사를 하나의 거대한 JS 객체에 넣는다.
- 모든 챕터와 선택지를 초기 로딩 때 한꺼번에 불러온다.
- 조건을 확인할 때 본문 텍스트까지 같이 읽는다.
- 이벤트 하나를 고치기 위해 엔진 코드 파일을 수정해야 한다.
- 선택지 추가가 코드 변경과 동일한 위험도를 갖는다.

좋은 방향:
- 엔진 코드는 상태, 조건 검사, 로더, 렌더러, 효과 처리만 담당한다.
- 스크립트 본문은 JSON, JSONL, YAML, Markdown+metadata 등 외부 데이터 파일로 둔다.
- 선택지, 플래그 변경, 다음 이벤트 이동, 효과 지시는 데이터에 선언한다.
- manifest에는 본문이 아니라 id, 파일 경로, trigger, 조건 요약, 우선순위만 둔다.
- 실제 본문은 조건을 통과한 뒤에만 로드한다.

결론:
스크립트는 코드가 아니라 데이터다.
코드는 데이터를 실행하는 무대 장치여야 한다.

============================================================
3. OFF/ON 호출 모델
============================================================

대부분의 스크립트는 기본적으로 OFF 상태다.
OFF는 “파일이 존재하지만 현재 실행 대상이 아니며, 메모리에 올라오지 않았거나 출력되지 않는 상태”를 뜻한다.

흐름은 다음과 같다.

1. 게임 상태가 존재한다.
   예: 현재 위치, 현재 챕터, 플래그, 변수, 호감도, 공포 수치, 방문 여부, 선택 이력.

2. 사용자가 행동하거나 상태가 변한다.
   예: 장소 이동, 선택지 클릭, 시간 경과, 특정 플래그 활성화, 아이템 획득, 이벤트 종료.

3. 해당 상태 변화에 대응하는 triggerKey를 만든다.
   예: location:burned_chapel, flag:entered_chapel, choice:inspect_altar.

4. manifest에서 그 triggerKey에 연결된 후보 이벤트만 가져온다.

5. 후보 이벤트의 조건만 검사한다.
   이때 검사 대상은 본문이 아니라 가벼운 조건 메타데이터다.

6. 조건을 통과한 이벤트만 ON 상태가 된다.

7. ON이 된 이벤트의 본문 파일을 로드한다.

8. 렌더러가 본문을 출력하고, 효과 엔진이 움직임/깜빡임/강조/지연 출력 등을 적용한다.

9. 이벤트 종료 후 상태를 갱신한다.
   예: 완료 플래그 설정, 다음 이벤트 예약, 캐시 유지 또는 해제.

10. 이벤트는 다시 OFF 또는 완료 상태로 돌아간다.

핵심:
많은 스크립트가 존재해도 항상 켜져 있으면 안 된다.
조건을 만족한 소수의 스크립트만 잠깐 켜져야 한다.

============================================================
4. manifest/index의 역할
============================================================

manifest는 대용량 스크립트 시스템의 카드목록이다.
manifest는 가능한 한 작고 가벼워야 한다.

manifest에 들어갈 정보:
- eventId
- script file path
- triggerKey
- chapter/scene/location
- 조건 요약
- 우선순위
- 중복 실행 가능 여부
- 완료 후 설정할 플래그
- 필요한 최소 상태 정보
- 프리로드 후보 여부

manifest에 넣지 말아야 할 정보:
- 긴 대사 본문
- 선택지 전체 문장
- 장문의 연출 서술
- 이미지/음향/효과 상세 데이터 전체
- 챕터 전체 텍스트

manifest의 목적:
- 현재 상황과 관련 없는 이벤트를 처음부터 검사 대상에서 제외한다.
- 전체 스크립트 목록을 매번 순회하지 않게 한다.
- 조건 검사 전에 필요한 후보군을 매우 작게 줄인다.
- 로딩할 파일 경로를 제공한다.
- Codex와 사람이 전체 구조를 빠르게 파악할 수 있게 한다.

중요:
대용량 프로젝트에서는 “전체 스크립트에서 조건 맞는 것 찾기”가 아니라
“현재 triggerKey에 등록된 후보 중 조건 맞는 것 찾기”가 되어야 한다.

============================================================
5. triggerKey 설계
============================================================

triggerKey는 조건 검사를 시작하는 문이다.
상태가 변할 때마다 전체 이벤트를 뒤지는 대신, 해당 변화와 관련된 key만 확인한다.

triggerKey 예시:
- location:burned_chapel
- location:rosenthal_gate
- flag:met_abel
- flag:entered_cathedral
- choice:inspect_coffin
- chapter:01:start
- chapter:01:end
- time:night
- item:holy_relic_acquired
- state:fear_changed
- ui:open_journal
- event:chapel_001_done

좋은 triggerKey의 조건:
1. 너무 넓지 않아야 한다.
   예: all, any, update 같은 key는 위험하다.

2. 너무 세분화되어 관리 불가능해지면 안 된다.
   key가 지나치게 많아지면 사람이 구조를 잃는다.

3. 실제 게임 상태 변화와 대응되어야 한다.

4. 하나의 이벤트가 여러 triggerKey에 등록될 수는 있다.
   단, 중복 실행 방지 규칙이 필요하다.

5. triggerKey는 manifest에서 빠르게 후보를 찾기 위한 색인이다.

중요:
triggerKey는 “이 상황에서 후보가 될 수 있는 이벤트 묶음”을 찾는 장치다.
최종 실행 여부는 condition gate가 결정한다.

============================================================
6. condition gate / 조건 게이트
============================================================

condition gate는 후보 이벤트를 실제로 실행할지 판단하는 검문소다.
조건 검사는 반드시 가벼워야 한다.

조건에 사용할 수 있는 것:
- 현재 챕터
- 현재 장소
- 활성 플래그
- 비활성 플래그
- 수치 변수
- 선택 이력
- 이벤트 완료 여부
- 인물 상태
- 아이템 보유 여부
- 우선순위
- 랜덤 여부가 필요할 경우 seed 기반 규칙

조건 검사에서 피해야 할 것:
- 긴 텍스트 본문 검색
- 대용량 파일 로딩
- 모든 챕터 전체 탐색
- DOM 상태를 과도하게 읽기
- 비동기 fetch를 조건마다 반복
- 매 프레임 전체 조건 검사

조건 게이트의 원칙:
1. 조건은 상태 객체와 manifest 메타데이터만 보고 판단한다.
2. 조건 통과 전에는 스크립트 본문 파일을 로드하지 않는다.
3. 조건 검사 함수는 순수 함수에 가깝게 유지한다.
4. 조건 실패는 정상 흐름이다. 오류로 취급하지 않는다.
5. 조건 통과 이벤트가 여러 개일 경우 우선순위 규칙이 필요하다.

============================================================
7. 이벤트 우선순위와 충돌 처리
============================================================

동일한 triggerKey에서 여러 이벤트가 조건을 통과할 수 있다.
이때 실행 규칙이 없으면 예측 불가능한 결과가 생긴다.

필요한 규칙:
1. priority 값이 높은 이벤트를 먼저 실행한다.
2. exclusive 이벤트는 같은 trigger에서 다른 이벤트와 동시에 실행하지 않는다.
3. repeatable=false인 이벤트는 완료 후 다시 실행되지 않는다.
4. queue 가능한 이벤트는 순서대로 대기열에 넣는다.
5. background 이벤트와 foreground 이벤트를 구분한다.
6. 강제 이벤트, 선택 이벤트, 장식 이벤트를 구분한다.

권장 분류:
- main: 진행에 필수인 메인 이벤트
- side: 조건부 사이드 이벤트
- flavor: 짧은 분위기 텍스트
- effect: 화면 효과 중심 이벤트
- system: 저장/상태 갱신용 내부 이벤트
- debug: 개발 중 확인용 이벤트

충돌 처리 원칙:
- 스토리 진행을 막는 이벤트가 최우선이다.
- 이미 실행된 일회성 이벤트는 재실행하지 않는다.
- 같은 화면에서 텍스트 이벤트와 강한 화면 효과가 충돌하지 않게 한다.
- 조건이 겹치는 이벤트는 manifest에서 의도적으로 정렬한다.

============================================================
8. 데이터 파일 분할 원칙
============================================================

대용량 스크립트는 한 파일에 몰아넣으면 안 된다.
특히 all_scripts.json 같은 거대한 파일은 피해야 한다.

권장 분할 기준:
- chapter
- scene
- location
- event type
- character
- route
- system/event/effect 분리

예시 구조:
- /data/manifest.json
- /scripts/chapter01/chapel_001.json
- /scripts/chapter01/chapel_002.json
- /scripts/chapter01/abel_001.json
- /scripts/chapter02/rosenthal_gate_001.json
- /scripts/common/flavor_rain.json
- /scripts/system/tutorial_flags.json

권장 파일 크기:
- manifest는 가능한 작게 유지한다.
- 본문 스크립트 파일은 너무 크지 않게 나눈다.
- 한 장면/이벤트 단위 파일은 사람이 읽고 수정 가능한 크기를 유지한다.
- 거대한 챕터 하나를 한 파일에 넣기보다 scene 단위로 쪼갠다.

이유:
- 작은 파일은 조건 통과 후 빠르게 로드된다.
- Codex가 한 번에 읽고 수정하기 쉽다.
- Git diff가 깨끗해진다.
- 충돌 해결이 쉬워진다.
- 한 이벤트 수정이 전체 챕터 파손으로 이어지지 않는다.

============================================================
9. 로딩 전략
============================================================

대용량 스크립트 시스템은 로딩 전략이 중요하다.

기본 원칙:
1. 초기 로딩에는 엔진 코드와 manifest만 불러온다.
2. 현재 챕터나 현재 위치와 무관한 본문은 로드하지 않는다.
3. 조건을 통과한 이벤트 본문만 fetch/load한다.
4. 한 번 읽은 파일은 캐시에 둘 수 있다.
5. 메모리 사용량이 커지면 오래된 캐시를 해제한다.
6. 다음에 사용할 가능성이 높은 파일만 조용히 preload한다.

프리로드 후보:
- 현재 이벤트의 next 이벤트
- 현재 장소에서 가능한 주요 이벤트
- 현재 챕터의 바로 다음 장면
- 선택지로 곧 연결될 수 있는 소수의 이벤트

프리로드 금지:
- 전체 챕터 무조건 로딩
- 모든 루트 무조건 로딩
- 아직 조건상 도달 불가능한 후반부 로딩
- 모든 선택지 결과 미리 로딩

핵심:
초기 로딩은 얇게, 조건 통과 후 로딩은 정확하게, 프리로드는 적게.

============================================================
10. 캐시 전략
============================================================

스크립트 캐시는 성능을 높일 수 있지만, 무제한 캐시는 위험하다.

캐시할 수 있는 것:
- 방금 실행한 이벤트
- 현재 장면의 다음 이벤트
- 현재 위치에서 자주 호출될 flavor 이벤트
- 짧은 공통 효과 데이터

캐시에서 빼야 할 것:
- 오래 전에 끝난 챕터의 대용량 스크립트
- 현재 루트에서 더 이상 접근 불가능한 이벤트
- 큰 파일인데 재사용 가능성이 낮은 데이터

캐시 정책:
- 최근 사용 기준 LRU 방식 고려
- 챕터 이동 시 이전 챕터 캐시 해제
- 저장/로드 시 필요한 최소 상태만 저장
- 캐시에 저장된 본문과 게임 상태를 혼동하지 않는다.

중요:
캐시는 성능 장치이지 정본 저장소가 아니다.
정본 데이터는 파일에 있고, 현재 진행 상태는 state에 있다.

============================================================
11. 상태(state) 설계
============================================================

state는 현재 게임의 사실을 담는다.
스크립트 본문과 상태를 섞으면 안 된다.

state에 들어갈 것:
- 현재 챕터
- 현재 장소
- 현재 이벤트 id
- flags
- variables
- character states
- inventory
- choice history
- completed events
- route info
- display settings
- save metadata

state에 들어가면 안 되는 것:
- 대량 스크립트 본문
- 전체 manifest 복사본
- 모든 선택지 텍스트
- 렌더링된 DOM 조각
- 임시 애니메이션 상태 전체

원칙:
- state는 작고 직렬화 가능해야 한다.
- 저장 파일은 state 중심이어야 한다.
- 스크립트 파일은 저장 파일에 복사하지 않는다.
- 로드 시 state를 복원하고, state에 맞는 manifest 후보를 다시 찾는다.

============================================================
12. 렌더러와 효과 엔진
============================================================

렌더러는 스크립트 데이터를 화면으로 바꾸는 역할이다.
효과 엔진은 움직임, 깜빡임, 흔들림, 지연 출력 같은 연출을 적용한다.

중요 원칙:
1. 스크립트 데이터는 “무엇을 출력할지”를 선언한다.
2. 렌더러는 “어떻게 화면에 넣을지”를 담당한다.
3. 효과 엔진은 “어떤 시각적 변화가 붙을지”를 담당한다.
4. 스크립트 본문이 직접 DOM을 조작하게 하지 않는다.
5. 효과는 데이터로 선언하고 엔진이 해석한다.
6. 과한 DOM 노드 생성과 layout/reflow 반복을 피한다.

효과 적용 시 주의:
- 글자별 DOM 분리는 매우 무거울 수 있다.
- 모든 글자를 span으로 감싸고 계속 움직이면 렉이 생긴다.
- 큰 텍스트에는 효과를 제한적으로 적용한다.
- 화면에 보이는 부분에만 효과를 적용한다.
- 깜빡임/흔들림/부유 효과는 CSS transform/opacity 중심으로 처리하는 것이 좋다.
- 레이아웃을 자주 바꾸는 top/left/width/height 변경은 피한다.
- requestAnimationFrame을 남발하지 않는다.
- 애니메이션은 종료 후 정리한다.

핵심:
텍스트가 많을수록 효과는 더 작고 조심스럽게 적용해야 한다.
대용량 스크립트에서 가장 위험한 병목은 데이터 양보다 DOM 폭발일 수 있다.

============================================================
13. 조건 검사의 성능 원칙
============================================================

조건 검사 자체는 보통 가볍다.
느려지는 이유는 조건 검사 중에 무거운 데이터를 같이 만지기 때문이다.

피해야 할 것:
- 조건마다 파일 fetch
- 조건마다 JSON parse
- 조건마다 전체 텍스트 검색
- 조건마다 DOM 조회
- 조건마다 전체 이벤트 배열 순회
- 매 프레임 전체 조건 검사

해야 할 것:
- triggerKey로 후보를 줄인다.
- 후보 조건만 검사한다.
- 조건은 flags/variables 중심으로 본다.
- 본문 로딩은 조건 통과 후에 한다.
- 조건 검사 결과를 필요하면 짧게 캐시한다.
- 상태가 바뀐 순간에만 관련 trigger를 확인한다.

중요:
매 프레임 검사하지 않는다.
장소 이동, 선택지 클릭, 플래그 변화, 이벤트 종료 같은 상태 변화 지점에서만 검사한다.

============================================================
14. Codex 작업 지침
============================================================

Codex는 이 문서를 읽고 다음 원칙을 따라야 한다.

1. 대용량 스크립트 본문을 엔진 코드에 직접 넣지 말 것.
2. manifest/index 기반의 조건부 호출 구조를 우선 고려할 것.
3. 전체 스크립트 순회 구조를 만들지 말 것.
4. 조건 검사 전에 본문 파일을 로드하지 말 것.
5. 조건 검사와 렌더링과 효과 적용을 분리할 것.
6. state와 script data를 분리할 것.
7. 저장 데이터에는 진행 상태만 넣고, 스크립트 본문은 복사하지 말 것.
8. 큰 파일 하나보다 작은 장면 단위 파일을 선호할 것.
9. 렌더링 최적화 없이 글자별 효과를 대량 적용하지 말 것.
10. 기존 프로젝트를 수정할 때는 먼저 현재 구조를 요약하고, 변경 범위를 제안한 뒤 진행할 것.
11. 구현 전에는 파일 구조 제안, 데이터 흐름, 성능 위험, 마이그레이션 순서를 먼저 작성할 것.
12. 사용자가 “이론만” 요청한 경우 실제 코드를 대량 작성하지 말 것.

============================================================
15. 권장 아키텍처 개념
============================================================

엔진 레이어:
- state manager
- condition evaluator
- event dispatcher
- script loader
- cache manager
- renderer
- effect manager
- save/load manager

데이터 레이어:
- manifest/index
- script files
- choice files
- effect declarations
- localization/text data if needed

흐름:
1. 사용자가 행동한다.
2. state가 변한다.
3. event dispatcher가 triggerKey를 만든다.
4. manifest에서 후보 이벤트 목록을 가져온다.
5. condition evaluator가 후보를 필터링한다.
6. 우선순위와 충돌 규칙으로 실행 이벤트를 정한다.
7. script loader가 해당 파일만 로드한다.
8. renderer가 텍스트와 선택지를 화면에 출력한다.
9. effect manager가 선언된 효과를 적용한다.
10. 이벤트 완료 후 state를 갱신한다.
11. 필요한 경우 다음 trigger를 예약한다.

============================================================
16. 대용량 프로젝트에서 특히 금지할 것
============================================================

금지:
- all_scripts.json 하나에 전체 스토리 몰아넣기
- main.js에 대사/선택지 직접 삽입
- 매 프레임 모든 이벤트 조건 검사
- 조건 검사 중 fetch/parse 반복
- 화면 밖 텍스트까지 글자별 애니메이션 적용
- 이미 완료된 이벤트 반복 검사
- 저장 파일에 스크립트 본문 저장
- manifest에 장문 본문 삽입
- 한 번의 Codex 작업으로 전체 구조를 갈아엎기
- “알아서 다 최적화” 같은 넓은 명령으로 대규모 수정하기

권장:
- 작은 단위 작업
- 파일 단위 diff 확인
- manifest 먼저 설계
- 조건 게이트 먼저 설계
- 이후 로더/렌더러/효과 엔진 연결
- 성능 테스트는 데이터가 적을 때부터 준비
- 선택지/스크립트 데이터 검증 도구 마련

============================================================
17. 검증 도구 필요성
============================================================

대용량 스크립트는 사람이 눈으로만 검수하면 깨진다.
따라서 데이터 검증 도구가 필요하다.

검증해야 할 것:
- manifest에 등록된 file path가 실제 존재하는가
- eventId가 중복되지 않는가
- next로 연결된 이벤트가 존재하는가
- choice가 가리키는 next가 유효한가
- 필요한 flag 이름이 오타 없이 쓰였는가
- 완료 플래그와 재실행 규칙이 충돌하지 않는가
- 조건상 절대 도달 불가능한 이벤트가 있는가
- 순환 참조가 의도된 것인지 확인 가능한가
- 효과 선언이 renderer/effect manager가 지원하는 타입인가
- 대용량 파일이 기준 크기를 넘지 않는가

Codex는 구현 단계에서 가능하면 데이터 검증 스크립트를 별도로 제안해야 한다.

============================================================
18. 요약
============================================================

이 프로젝트의 대용량 스크립트 구조는 다음 원칙을 따른다.

1. 스크립트는 기본 OFF.
2. 조건 만족 시에만 ON.
3. ON 전에 본문 로드 금지.
4. manifest는 가벼운 색인.
5. triggerKey로 후보를 먼저 줄임.
6. condition gate는 상태와 메타데이터만 검사.
7. 본문은 조건 통과 후 필요한 파일만 로드.
8. 렌더러와 효과 엔진은 데이터 선언을 해석.
9. 스크립트, 상태, 렌더링, 효과는 분리.
10. 대용량 텍스트보다 DOM 폭발이 더 위험할 수 있음.
11. 모든 스크립트 검사 금지.
12. 모든 파일 한 번에 로딩 금지.
13. 작은 파일, 작은 후보군, 작은 변경 단위 유지.
14. Codex는 구현 전 구조 요약과 변경 계획을 먼저 제시해야 함.

최종 문장:
코드는 작게, 데이터는 나누고, 조건은 가볍게, 호출은 정확하게, 출력은 보이는 만큼만.
